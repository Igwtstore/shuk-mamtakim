🚀 ¿Mudamos el sistema a una base de datos real?

De Google Apps Script + Sheets → Supabase · pros, contras y plan
Informe para Jony · Junio 2026 · Uso interno
En 30 segundos: Hoy el sistema corre sobre una "planilla gigante en la nube" (Google Sheets) manejada por Apps Script. Es gratis y cómodo, pero lento y no escala. Mudarlo a Supabase (una base de datos de verdad, la que ya usás para el login) te da velocidad y escala — el precio es trabajo de mudanza (semanas) y perder la comodidad de editar a mano en la planilla. No es urgente hoy; es el paso para vender el sistema en serio.

1. Qué tenés hoy (y por qué arrancamos así)

Tu sistema son tres piezas:

PiezaQué hace
Google SheetsEs la base de datos: ahí viven productos, ventas, pagos, stock. Es una planilla.
Apps ScriptEs el motor (backend): lee/escribe la planilla y le contesta a la página.
VercelSirve la página (lo que ves). Esto NO se toca, queda igual.

Por algo arrancamos así, y fue la decisión correcta para empezar:

✅ $0 de costo. Cero.  ·  ✅ Cero infraestructura que mantener (Google se encarga).  ·  ✅ Ves y editás los datos a mano en la planilla (comodísimo para corregir o cargar saldos viejos del Excel).  ·  ✅ Backup automático a Drive.  ·  ✅ Arrancaste sin contratar nada y validaste que el sistema sirve.

Para nacer y probar, no había nada mejor. Pero tiene un techo, y lo estás tocando.

2. El problema (lo que te pasó hoy)

SíntomaCausa de fondo
"No se pudo cargar"Apps Script es un motor compartido y gratuito: cada consulta arranca "en frío" y tarda 1 a 5 segundos, y Google limita cuántas podés hacer a la vez.
Todo lento al abrir el panelGoogle Sheets no es una base de datos, es una planilla. Leerla entera es lento, y el panel pide muchas cosas juntas.
No aguantaría varias tiendasSi mañana vendés el sistema a 5 negocios usando a la vez, Sheets + Apps Script se satura.

3. A dónde se mudaría: Supabase

Supabase es una base de datos profesional (PostgreSQL) en la nube, con login, almacenamiento de archivos y API automática. Ya la estás usando para que entres al panel. La idea es que también guarde los datos (productos, ventas, pagos…).

4. ¿Hay otras opciones? Sí — y por qué Supabase

Supabase no es la única. Estas son las alternativas reales:

OpciónQué esMantenimiento¿Como tus planillas?
Supabase ⭐Base de datos SQL (tablas) + login + archivos, todo incluido. Código abierto.Casi ceroSí (filas y columnas)
Firebase (Google)Lo mismo, pero de Google. Guarda "documentos" (otro modelo más raro).Casi ceroNo

⚠️ Ojo con el "servidor dedicado": suena bien, pero es más trabajo, no menos. Ahí vos pasás a ser el encargado del servidor (seguridad, copias de respaldo, que no se caiga). Hoy de todo eso se encarga Google gratis. Por eso conviene un servicio "todo incluido".

Por qué Supabase entre todas

  • Es SQL = tablas con filas y columnas → lo más parecido a las planillas que ya manejás (Firebase usa un modelo más difícil de entender).
  • Código abierto = tus datos son TUYOS y exportables. No quedás "preso" de nadie (con Firebase quedás atado a Google).
  • Ya la usás para el login → una sola cuenta para todo, menos piezas.

Ninguna te ata para siempre — los datos siempre se pueden mover a otra.

6. Lo que GANÁS y lo que PERDÉS (lo que pediste)

✅ Lo que GANÁS con Supabase

  • Velocidad: respuestas en milésimas de segundo en vez de 1-5 seg. Chau "No se pudo cargar".
  • Aguanta gente a la vez: varias tiendas/usuarios sin caerse → clave para vender.
  • Escala de verdad: miles de ventas y se filtra/busca al instante.
  • Datos sólidos: una base de datos no se "rompe" si alguien toca mal una celda.
  • Imagen profesional: un sistema vendible no corre sobre una planilla.

❌ Lo que PERDÉS / el costo

  • Editar a mano en la planilla: hoy abrís el Sheet y corregís. Eso se pierde — todo pasa a hacerse desde pantallas. (Mitigable: te puedo hacer una "vista de tabla" para editar masivo, que ya casi tenés con el editor masivo.)
  • Costo: Supabase es gratis hasta ~500 MB y 50.000 usuarios/mes. Para tu volumen, gratis te alcanza un buen tiempo. Después, plan Pro ~US$ 25/mes.
  • Trabajo de mudanza: diseñar la base, copiar los datos y reescribir el motor. Semanas de laburo (mío), no un fin de semana.
  • Otro proveedor del que dependés (aunque Supabase es código abierto y los datos son exportables — no quedás "preso").
  • El backup cambia: ya no es "a Drive" sino el de Supabase (automático, pero distinto a lo que conocés).

8. Cómo se haría (sin romper nada)

0
Diseñar la base. Definir las tablas (productos, ventas, pagos, stock…) — el espejo de tus hojas de hoy.
1

🎯 Mi recomendación honesta

No migres todavía. El sistema funciona y la lentitud, aunque molesta, se baja optimizando el panel (gratis, lo hago ya).

La migración es EL paso para vender el sistema (white-label). El día que arranques en serio con varias tiendas, Sheets no aguanta y Supabase sí — ahí se justifica el trabajo de semanas y el ~US$25/mes.

Plan sugerido: 1) ahora optimizo el panel + nivelo las opciones de plata; 2) seguimos con el white-label sobre Sheets; 3) cuando haya 2-3 tiendas reales o sientas el techo, migramos a Supabase con el plan de arriba. Vos decidís el cuándo — yo te dejo el camino listo.
